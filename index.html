// MODULE 01 — COVERAGE

Coverage in
SystemVerilog

Coverage is the metric that tells you how much of your design intent has been exercised by your testbench.

// INSTRUCTOR NOTE
Think of coverage as a GPS tracker for your tests. You write stimulus, but how do you know you've hit all the important scenarios? Coverage gives you that answer. Without it, you're flying blind — your simulation might pass but miss critical corner cases that only show up in silicon.
📐
Code Coverage
Structural coverage automatically measured by the simulator tool. No user code needed.
// Types:
line → every line executed?
toggle → 0→1, 1→0 on signals
branch → if/else paths taken
expr → boolean sub-exprs
fsm → state transitions
🎯
Functional Coverage
User-defined coverage written in SystemVerilog. Captures design intent not structure.
covergroup cg;
  coverpoint addr;
  cross addr, rw;
endgroup

// LESSON 01
COVERGROUP
Basic covergroup
A covergroup is a user-defined type that groups related coverage points together. Instantiate it like a class.
SV
covergroup addr_cg @(posedge clk);
  // coverpoint samples 'addr' on posedge clk
  cp_addr : coverpoint addr {
    bins low = {[0:63]};
    bins mid = {[64:191]};
    bins high = {[192:255]};
  }
endgroup

// Instantiate
addr_cg cg_inst = new();
// Call manually if no clk event
cg_inst.sample();
covergroup — defines the coverage container type. Must be instantiated with new()
coverpoint — the signal or expression being sampled
bins — partitions the coverpoint's value range into named buckets
@(posedge clk) — auto-samples at every rising edge. Remove to sample manually via .sample()
// LESSON 02
BINS
Types of Bins
Bins control how values are grouped. Illegal and ignore bins shape what gets counted.
SV
covergroup opcode_cg;
  coverpoint opcode {
    // explicit — each value = 1 bin
    bins read = {4'hA};
    bins write = {4'hB};

    // wildcard bin
    wildcard bins ctrl = {4'b11??};

    // transition bin: A then B
    bins rd2wr = (4'hA => 4'hB);

    // exclude from coverage
    ignore_bins rsvd = {4'hF};

    // mark as error if hit
    illegal_bins bad = {4'h0};
  }
endgroup
bins — counts when this value/range is sampled
wildcard bins — treats ? as don't-care (X or Z)
transition bin (=>) — hit only when value sequence occurs in order
ignore_bins — excluded from coverage metric, not an error
illegal_bins — triggers a simulation error if hit
// LESSON 03
CROSS
Cross Coverage
cross measures combinations of two or more coverpoints — catches scenarios missed by individual coverpoints.
SV
covergroup bus_cg;
  coverpoint rw {
    bins read = {0};
    bins write = {1};
  }
  coverpoint size {
    bins byte_ = {0};
    bins hword = {1};
    bins word_ = {2};
  }
  // 2 x 3 = 6 cross bins
  cross rw, size;
endgroup
cross — creates N×M bins from two coverpoints. Checks every combination (e.g. "read 4-byte", "write 2-byte") was exercised
Without cross, 100% on each coverpoint still wouldn't tell you if "write + word" was ever tested together
Use ignore_bins inside cross to exclude impossible combos
// LESSON 04
OPTIONS
covergroup Options
Fine-tune sampling behaviour, goals, and bin creation using the option namespace.
SV
covergroup cfg_cg;
  // set coverage goal (default 100%)
  option.goal = 90;

  // per-instance naming
  option.name = "bus_cg_inst";

  // auto-bin count (when no explicit bins)
  option.auto_bin_max = 8;

  // count only first N hits per bin
  option.at_least = 10;

  coverpoint data;
endgroup
option.goal — what % counts as "passing" for this group
auto_bin_max — when no explicit bins, tool creates up to N bins automatically
option.at_least — bin is only "covered" after being hit N times, not just once
type_option vs option — type_option applies to the class, option applies to each instance
// MODULE 03 — ASSERTIONS (SVA)

SystemVerilog
Assertions (SVA)

SVA lets you formally describe temporal behaviour of your design and automatically check it every clock cycle.

// TYPE 01
IMMEDIATE
Immediate Assertions
Procedural — checked instantly at the moment of execution, like an if-statement. No clock needed.
SV
always @(posedge clk) begin
  // basic immediate assert
  assert (req !== 1'bx)
    else $error("req is X!");

  // with pass action block
  assert (valid && data != 0) begin
    $info("valid data seen");
  end else begin
    $error("data invalid!");
  end

  // assume — constrain in formal
  assume (mode inside {0,1,2});
end
assert — evaluates expression NOW. Fails immediately if false
Pass block (before else) runs on success; else block runs on failure
assume — same syntax, used in formal verification to constrain inputs
// TYPE 02
CONCURRENT
Concurrent Assertions
Clock-aware — checks a temporal property across multiple clock edges. The real power of SVA.
SV
// Sequence: req HIGH, then ack within 1-3 cycles
sequence req_ack_seq;
  @(posedge clk)
  req ##1 ack[*1:3];
endsequence

// Property: whenever req, ack follows
property req_ack_p;
  @(posedge clk)
  disable iff (rst)
  req |=> ack;
endproperty

// Bind assertion to property
ASSERT_REQ_ACK: assert property(req_ack_p)
  else $error("ACK missing!");
sequence — defines a temporal pattern using operators like ##, [*], etc.
property — wraps a sequence with a clock, disable condition, and implication
assert property — checks the property every clock cycle, flags failure
disable iff — vacuously passes (assertion disabled) when condition is true

// MODULE 04 — TEMPORAL OPERATORS

Sequence Operators

The building blocks of all concurrent assertions. Master these and you can describe any protocol behaviour.

OPERATOR SYNTAX MEANING EXAMPLE
##N a ##2 b b must be true exactly N cycles after a req ##1 ack — ack follows req by 1 cycle
##[m:n] a ##[1:4] b b occurs between m and n cycles after a start ##[2:5] done
[*N] a[*3] a is true for exactly N consecutive cycles valid[*4] — valid high for 4 cycles
[*m:n] a[*1:4] a is true for m to n consecutive cycles hold[*2:8]
[+] a[+] a is true for 1 or more cycles (= [*1:$]) busy[+]
[*] a[*] a is true for 0 or more cycles (= [*0:$]) idle[*]
[->N] a[->2] Non-consecutive — a true exactly N times, last occurrence ends match req[->3]
[=N] a[=2] Non-consecutive — a true N times, with anything after last occurrence grant[=2]
|=> a |=> b Overlapping implication — if antecedent matches, check b starting NEXT cycle req |=> ack
|-> a |-> b Non-overlapping implication — check b in same cycle antecedent ends valid |-> !err
first_match first_match(seq) Matches on first possible completion of a sequence Useful with multi-length repeats
throughout a throughout seq a must remain true during the entire sequence en throughout (req ##1 ack)
within s1 within s2 s1 must occur within the time frame of s2 Containment checking
intersect s1 intersect s2 Both sequences start and end at same time Parallel protocol checking
// MODULE 05 — KEYWORD REFERENCE

All SVA & Coverage
Keywords Explained

assert
Check a condition — fail if false
Used both as immediate (assert(expr)) and concurrent (assert property(p)). Simulation reports error on failure. Named assertions help with debugging.
assume
Constrain inputs for formal verification
In simulation: same as assert. In formal: tells the tool to only consider inputs that satisfy this. Used to restrict state space to legal scenarios.
cover
Measure how often a property/sequence is hit
cover property(p) — records every time the property is true. Unlike assert, failing to hit it doesn't error; it shows up as 0% coverage.
property
Named, reusable temporal specification
Wraps a sequence with a clock, optional disable iff, and logical operators. Properties can be composed from other properties. Used with assert/assume/cover.
sequence
Temporal pattern of signal behaviour
Describes a multi-cycle event pattern using ##, [*], |=>, etc. Sequences can be instantiated inside properties or other sequences. They are the building blocks of SVA.
disable iff
Vacuous pass condition for reset/enable
When the condition is true (e.g. reset), the property is disabled — it neither passes nor fails. Critical for avoiding false failures during reset sequences.
covergroup
User-defined coverage container type
Defined like a class, instantiated with new(). Can be triggered by a clock event or sampled manually. Groups related coverpoints measuring design intent.
coverpoint
Signal or expression to be measured
Each coverpoint creates bins automatically (or explicitly). Must be inside a covergroup. Coverage = (bins hit / total bins) × 100%.
bins
Named value bucket inside a coverpoint
A coverpoint is "covered" when all its bins are hit. You can define value bins {0,1,2}, range bins [0:255], or transition bins (A => B).
ignore_bins
Exclude values from coverage metric
Values in ignore_bins don't count for or against coverage — they're simply not tracked. Use for reserved/unused encodings that shouldn't affect your metric.
illegal_bins
Report error if this value is sampled
Stronger than ignore — if a value matching illegal_bins is sampled, the simulator raises an error. Use to catch protocol violations like reserved opcode usage.
cross
Cross-product of two or more coverpoints
Creates N×M bins for every combination. Ensures all combinations of two signals are tested — catches corner cases that per-signal coverage misses entirely.

Advanced SVA Keywords

KEYWORD CATEGORY DESCRIPTION & EXAMPLE
$rose(sig) Sampled Value Fn True when sig transitions from 0 to 1. $rose(req) |=> ack
$fell(sig) Sampled Value Fn True when sig transitions from 1 to 0.
$stable(sig) Sampled Value Fn True when sig did NOT change from previous cycle.
$past(sig, N) Sampled Value Fn Returns value of sig N cycles ago. Default N=1. $past(data) == data
$onehot(bus) System Function True when exactly one bit of bus is high. Great for grant bus checks.
$countones(bus) System Function Returns number of 1-bits in bus.
not property(p) Property Operator Inverts a property. assert property(not (a ##1 b)) — a followed by b must NOT happen.
s_eventually LTL Operator Strong eventually — the property MUST occur at some point. Used in formal.
bind Instantiation Attaches assertions to a module without modifying it. bind dut my_sva_module inst(.*)
// MODULE 06 — COMPLETE PROTOCOL EXAMPLES

Real-World Examples

01
AXI Handshake Assertion
AXI spec: once VALID is asserted, it must stay high until the handshake (VALID & READY). Also, READY response must come within a max number of cycles.
PROTOCOL EXAMPLE
// Rule 1: VALID must not deassert before READY
property valid_stable;
  @(posedge clk) disable iff(rst)
  (VALID && !READY) |-> ##1 VALID;
endproperty
AXI_VALID_STABLE: assert property(valid_stable);

// Rule 2: READY must come within 16 cycles of VALID
property ready_timeout;
  @(posedge clk) disable iff(rst)
  VALID |=> READY[->1] within ##[1:16] VALID;
endproperty
AXI_READY_TIMEOUT: assert property(ready_timeout);

// Cover: measure successful handshakes
AXI_HANDSHAKE_HIT: cover property(
  @(posedge clk) VALID && READY
);
02
Complete Functional Coverage: AXI Transaction
Capture all meaningful combinations of AXI transaction attributes — burst type, size, length — to ensure comprehensive protocol coverage.
FUNCTIONAL COVERAGE
covergroup axi_cg @(posedge clk);

  // Burst type: FIXED, INCR, WRAP
  cp_burst: coverpoint AWBURST {
    bins fixed = {2'b00};
    bins incr = {2'b01};
    bins wrap = {2'b10};
    illegal_bins rsvd = {2'b11};
  }

  // Burst size: 1,2,4,8,16,32,64,128 bytes
  cp_size: coverpoint AWSIZE;

  // Burst length: 1-16 for FIXED, 1-256 for INCR
  cp_len: coverpoint AWLEN {
    bins len1 = {0};
    bins len4 = {3};
    bins len16 = {15};
    bins other = default;
  }

  // Cross: all burst_type x size combos
  cross cp_burst, cp_size, cp_len;

endgroup

// Instantiate in UVM monitor
axi_cg cg = new();
// In monitor write_item task:
cg.sample();
// INTERVIEW TIP 🎯
At AMD/Qualcomm/Intel interviews, they'll ask: "What's the difference between |-> and |=>". Answer: |-> is overlapping — the consequent is checked in the SAME cycle the antecedent ends. |=> is non-overlapping — consequent checked in the NEXT cycle. a |=> b is equivalent to a ##1 b |-> b.
